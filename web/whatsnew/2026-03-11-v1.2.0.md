7개에서 **8개 Gateway**로 분리하고 배포 안정성을 대폭 개선한 릴리스입니다. Container 응답이 **54% 빨라지고**(50s → 23s), 전체 테스트 **104/104 통과**를 달성했습니다.

:::tip 핵심 성과
**8 Gateway** · **10 Routes** · **104/104 테스트 통과** · **93% 분류 정확도** · **Container 54% 속도 개선**
:::

## Gateway 분리: Infra → Network + Container

v1.1.0의 Infra Gateway(41개 도구)가 너무 많은 도구를 포함하고 있어, 네트워크와 컨테이너 도메인으로 분리했습니다.

<GatewaySplitFlow />

| 항목 | Before (Infra 단일) | After (분리) |
|------|---------------------|-------------|
| Gateway | 1 (Infra, 41 도구) | 2 (Network 17 + Container 24) |
| Container 응답 시간 | ~50초 | ~23초 (**54% 개선**) |
| 도구 선택 정확도 | LLM이 41개 중 선택 | 24개 중 선택 |
| 라우트 체계 | 7 Gateway, 9 Routes | **8 Gateway, 10 Routes** |

:::info 왜 분리가 효과적인가?
LLM이 41개 도구 설명을 모두 읽고 선택하는 것보다, 17개(Network) 또는 24개(Container) 중에서 선택하는 것이 **더 빠르고 정확**합니다. 특히 Container 질문에서 네트워크 도구를 잘못 선택하는 오류가 사라졌습니다.
:::

## 배포 안정성 개선

배포 스크립트에 에러 핸들링과 자동 감지 로직을 추가했습니다.

| 스크립트 | 개선 사항 |
|----------|-----------|
| `06a-setup-agentcore-runtime.sh` | `run_or_fail` 헬퍼 + AWS 자격 증명 프리플라이트 체크 |
| `06b-setup-agentcore-gateway.sh` | 8 Gateway 일괄 생성 (Network/Container 분리 반영) |
| `07-setup-cloudfront-auth.sh` | CloudFront 자동 감지 — CDK 스택 우선, ALB origin 폴백 |
| Steampipe 연결 | 비밀번호 하드코딩 → `steampipe service status`에서 동적 조회 |

## 주요 버그 수정

| 이슈 | 원인 | 수정 |
|------|------|------|
| EKS access entry 실패 | `arn:aws:sts::` 형식 사용 | `arn:aws:iam::` 형식으로 변환 |
| Cognito custom domain 오류 | `SupportedIdentityProviders` 누락 | 파라미터 추가 |
| AgentCore 권한 오류 | 권한 부족 | `bedrock-agentcore:*` 권한 추가 |
| K8s PVC 조회 실패 | JSONB 직렬화 에러 | `k8s.ts`에서 `::text` 캐스팅 적용 |

:::warning Steampipe JSONB 주의
Kubernetes 테이블의 JSONB 컬럼(`spec`, `status`)은 직접 반환하면 직렬화 에러가 발생합니다.
`SELECT spec::text` 또는 `spec->>'key'`로 변환해야 합니다.
:::
